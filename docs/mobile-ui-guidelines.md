# 移动端交互规范

这份规范用于约束 `apps/mobile` 的业务页面设计，避免页面级导航、实体卡片和表单操作混在一起。

## 页面层级

- 列表页默认只展示列表、筛选、摘要和创建入口。
- 列表页不承载编辑、签约、退租、批量录入等长表单；选择列表项后最多展示摘要和操作入口。
- 列表项需要展示更多信息时，优先在被点击的卡片原地展开，避免把详情或操作面板放到列表底部造成上下文跳转。
- 详情页用于查看单个业务对象，先展示摘要信息，再展示该对象下的管理模块。
- 新建、编辑、批量录入、记录明细等表单默认不常驻显示，应由明确入口打开。
- 同一详情页存在多个轻量表单时，同一时间只展开一个表单。
- 当一个业务动作需要 3 个以上输入字段时，必须作为独立展开区、子页面或弹层处理，不能直接铺在列表或摘要区域。

## 操作放置

- 页面级导航放在页面内容顶部，不能放入实体卡片内部。
- 返回列表、返回详情、取消编辑属于页面级导航。
- 编辑当前实体、启用/停用当前条目、删除当前条目属于实体或条目级操作，可以放在对应卡片内。
- 保存、提交、确认添加属于表单级操作，应放在表单底部。
- 卡片内部不放“返回”按钮，除非该卡片本身是独立弹层或独立子页面容器。

## 表单展示

- 表单字段如果已有默认值，必须有固定标签，不能只依赖 placeholder 解释含义。
- 短字段成组展示时，每个字段都要有标签，例如“楼层数”“占地面积”“水费单价”。
- 详情页中的新增类表单通过入口按钮展开，入口按钮文案使用动作名称，例如“记录花费”“添加费用项”“批量添加”。
- 列表项的业务动作通过入口按钮打开表单，例如“编辑房间”“签约入住”；入口本身不能直接变成长表单。
- 表单提交成功后应自动收起，并回到详情展示状态。

## 卡片职责

- 列表卡片用于进入详情，只展示识别信息和关键摘要。
- 详情摘要卡片用于展示当前对象信息，可以放“编辑”这类对象级操作。
- 明细列表条目可以放启用/停用、选择、删除等条目级操作。
- 卡片不承担页面导航职责。

## 文案

- 按钮文案要说明动作结果，不使用含糊的“确定”“操作”。
- 返回按钮文案要说明返回目标，例如“返回公寓列表”“返回公寓详情”。
- 空状态文案说明下一步入口，例如“暂无公寓，点击新建开始维护”。
