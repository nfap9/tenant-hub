# Tenant Hub 功能需求文档（FRD）

> 本文档由 `docs/BUSINESS_STORIES.md` 逐条拆解而来。
> 采用抽象功能描述，不绑定具体页面实现与 API 设计。
> 后续系统功能对照检查以本文档为准。

> **映射说明：**
>
> - **页面**：对应 `docs/FRONTEND_PAGES.md` 中的 `PAGE-xxx`
> - **实现形式**：`独立页面` / `详情页内嵌弹窗` / `详情页内嵌组件` / `全局组件`
> - **API**：对应 `apps/api/src/routes/` 中的路由端点

---

## 零、系统基础功能

### 0.1 用户认证与鉴权

**FR-001 用户注册**

- 支持手机号注册，需验证短信验证码
- 支持设置登录密码
- 注册成功后自动创建个人账号

> **页面**：`PAGE-002` 注册页面（独立页面）  
> **API**：`POST /auth/otp`（发送验证码）、`POST /auth/register`（注册）

**FR-002 用户登录**

- 支持手机号+密码登录
- 支持手机号+短信验证码登录
- 登录成功后颁发访问令牌
- Token 需支持过期刷新机制
- 用户修改密码后，历史 Token 自动失效

> **页面**：`PAGE-001` 登录页面（独立页面）  
> **API**：`POST /auth/login/password`（密码登录）、`POST /auth/login/otp`（验证码登录）  
> **实现**：Token 刷新与失效校验由 `api/client.ts` 统一处理

**FR-003 密码管理**

- 支持用户修改登录密码（需校验旧密码）
- 支持通过短信验证码重置密码
- 密码需加密存储

> **页面**：`PAGE-003` 忘记密码页面（独立页面）；个人中心内嵌修改密码弹窗  
> **API**：`PUT /auth/password`（修改/重置密码）

**FR-004 用户信息管理**

- 用户可查看和修改个人基本信息（昵称、头像、联系方式）
- 用户可查看自己所属的组织列表

> **页面**：`PAGE-004` 个人中心页面（独立页面）  
> **API**：`GET /auth/me`（获取用户信息）、`PUT /auth/me`（更新用户信息）

**FR-005 会话管理**

- 支持查询当前活跃会话
- 支持主动退出登录（销毁当前会话）
- 支持在其他设备上强制下线

> **实现**：`AppSessionContext`（全局状态管理组件），`MainLayout`（组织切换弹窗）  
> **API**：`POST /auth/logout`（退出登录）

### 0.2 组织管理

**FR-006 组织创建**

- 用户可创建新组织，成为组织所有者
- 组织信息包括：名称、Logo、联系方式、地址

> **页面**：`PAGE-005` 组织管理页面（独立页面，含创建弹窗）  
> **API**：`POST /organizations`（创建组织）

**FR-007 组织成员管理**

- 邀请用户加入组织（通过手机号/链接）
- 查看组织成员列表及角色
- 移除组织成员
- 成员可主动退出组织

> **页面**：`PAGE-006` 组织成员管理页面（独立页面）  
> **API**：`GET /organizations/:id/members`（成员列表）、`DELETE /organizations/:id/members/:userId`（移除成员）  
> **注意**：当前前端为演示实现，角色变更和邀请功能待补齐真实 API

**FR-008 组织切换**

- 用户属于多个组织时，可在登录后切换当前工作组织
- 切换后所有业务数据按当前组织隔离

> **实现**：`MainLayout` 顶部导航栏组织下拉切换（全局组件）  
> **API**：`POST /organizations/:id/switch`（切换组织）

**FR-009 组织设置**

- 组织基本信息维护
- 组织级参数配置（如默认账单日、默认滞纳金费率等）

> **页面**：`PAGE-005` 组织管理页面（独立页面）  
> **API**：`GET /organizations/:id`（查询）、`PUT /organizations/:id`（更新）

### 0.3 系统通知

**FR-010 通知中心**

- 系统消息推送（合同到期提醒、账单到期提醒、工单提醒等）
- 通知列表查询与已读标记
- 支持短信、微信消息推送（预留）

> **页面**：`PAGE-007` 通知中心页面（独立页面）  
> **API**：`GET /notifications`（列表）、`PUT /notifications/:id/read`（标记已读）  
> **注意**：微信推送为预留功能，当前仅实现站内通知

## 一、物业管理模块

### 1.1 公寓（楼栋）管理

**FR-101 公寓基础信息管理**

- 维护公寓完整档案：名称、地址、物业类型、建筑年代、总层数、电梯数量
- 维护产权信息：产权性质、产权面积、公摊比例
- 维护房东合同信息：房东姓名、联系方式、合同起止日期、月租金、付款方式、递增规则
- 维护运营配置：水电成本单价、公摊比例、提醒日设置
- 维护消防与安全信息：消防等级、灭火器数量、逃生通道数量
- 支持按组织查询、搜索、筛选公寓

> **页面**：`PAGE-101` 公寓列表、`PAGE-102` 公寓新增/编辑、`PAGE-103` 公寓详情（均为独立页面）  
> **API**：`GET /apartments`（列表）、`POST /apartments`（创建）、`GET /apartments/:id`（详情）、`PUT /apartments/:id`（更新）、`DELETE /apartments/:id`（删除）

**FR-102 公寓状态生命周期管理**

- 状态定义：筹建中、装修中、待开业、运营中、停业整改、退租关闭
- 支持状态变更操作，变更时需记录原因和时间
- 状态变更约束：只有运营中状态允许签约
- 支持查询状态变更历史

> **页面**：`PAGE-104` 公寓状态变更（内嵌于 `PAGE-103` 公寓详情页的弹窗）  
> **API**：`PUT /apartments/:id/status`（状态变更）  
> **实现形式**：详情页「编辑」按钮旁的状态变更入口，弹窗选择新状态并填写原因

**FR-103 公寓运营支出管理**

- 记录公寓层面的各项支出：物业费、维修费、公共区域水电、电梯维保、消防检查、网络费等
- 支出按分类统计，支持月度/年度汇总
- 支出需关联到具体公寓，用于计算公寓级盈亏

> **页面**：`PAGE-105` 公寓运营支出页面（独立页面）  
> **API**：`GET /apartments/:id/expenses`（列表）、`POST /apartments/:id/expenses`（记录支出）

**FR-104 公寓可视化看板**

- 展示总房间数、已租数、空置数、维修中数量
- 展示本月应收、已收、欠收金额
- 展示入住率趋势（最近12个月）
- 展示租金单价分布

> **页面**：`PAGE-106` 公寓可视化看板（内嵌于 `PAGE-103` 公寓详情页的 Tab 组件 `ApartmentDashboard`）  
> **API**：`GET /apartments/:id/dashboard`（看板数据）、`GET /apartments/:id/occupancy-trend`（入住率趋势）、`GET /apartments/:id/rent-distribution`（租金分布）  
> **实现形式**：公寓详情页「经营看板」Tab，无独立路由页面

### 1.2 房间管理

**FR-201 房间档案管理**

- 维护房间基础信息：房间号、楼层、户型、朝向、面积
- 维护装修标准：毛坯/简装/精装/豪华装、装修日期
- 维护设施清单：床、衣柜、空调、热水器、洗衣机、冰箱、网络等（支持自定义）
- 维护房间状态及流转规则
- 状态变更约束（略，见原文）

> **页面**：`PAGE-107` 房间列表、`PAGE-108` 房间新增/编辑、`PAGE-110` 房间详情（均为独立页面）  
> **API**：`GET /apartments/rooms`（列表）、`POST /apartments/:id/rooms/batch`（创建）、`GET /apartments/rooms/:id`（详情）、`PUT /apartments/rooms/:id`（更新）、`DELETE /apartments/rooms/:id`（删除）  
> **注意**：后端无独立 `rooms.ts` 路由文件，房间 API 耦合在 `apartments.ts` 中

**FR-202 房间批量管理**

- 支持按楼层/户型批量创建房间
- 批量设置房间设施模板
- 批量调整租金定价

> **页面**：`PAGE-109` 房间批量创建页面（独立页面）  
> **API**：`POST /apartments/:id/rooms/batch`（批量创建）  
> **注意**：设施模板和租金批量调整当前仅支持创建时设置，后续批量编辑功能待补齐

**FR-203 房间维修工单管理**

- 工单来源：租客报修、运营巡检、退租检修
- 工单类型：水电、门窗、墙面、家具家电、网络、管道、其他
- 工单状态流转：待处理 → 已派单 → 处理中 → 待验收 → 已完成（或已取消）
- 工单信息：房间、报修人、联系电话、问题描述、优先级（紧急/一般）、预约上门时间、维修人员、材料费用、人工费用、完成照片、验收备注
- 紧急工单（漏水/断电）自动触发通知
- 维修费用按责任判定自动计入公寓运营支出或租客账单

> **页面**：`PAGE-112` 维修工单列表、`PAGE-113` 维修工单详情（均为独立页面）  
> **API**：`GET /maintenance`（列表）、`POST /maintenance`（创建）、`GET /maintenance/:id`（详情）、`PUT /maintenance/:id`（更新）

**FR-204 入住/退租检查清单管理**

- 入住检查：水电表底数、设施完好情况、钥匙数量、卫生状况
- 退租检查：设施损坏情况、卫生状况、需扣款项目
- 对比入住和退租状态，自动生成扣款建议
- 支持拍照留存
- 检查结果关联到租约和押金扣款

> **页面**：`PAGE-114` 检查清单（内嵌于 `PAGE-110` 房间详情页的组件 `RoomChecklistSection`）  
> **API**：`GET /room-checklists`（列表）、`POST /room-checklists`（创建）、`DELETE /room-checklists/:id`（删除）  
> **实现形式**：房间详情页「检查清单」区块，含表格、新建弹窗、详情弹窗、对比视图弹窗，无独立路由页面

### 1.3 房东合同管理

**FR-301 房东合同档案管理**

- 维护合同信息：合同编号、签约日期、起止日期、付款方式、租金金额、押金金额
- 租金递增规则：固定金额/百分比、递增周期
- 免租期设置
- 合同附件上传与管理
- 合同到期前30天、7天自动提醒

> **页面**：`PAGE-115` 房东合同列表、`PAGE-116` 房东合同新增/编辑、`PAGE-117` 房东合同详情（均为独立页面）  
> **API**：`GET /landlord-contracts`（列表）、`POST /landlord-contracts`（创建）、`GET /landlord-contracts/:id`（详情）、`PUT /landlord-contracts/:id`（更新）、`DELETE /landlord-contracts/:id`（删除）

**FR-302 房东付款计划管理**

- 根据合同自动生成付款计划（应付日期、金额）
- 记录实际付款（日期、金额、凭证号）
- 逾期未付预警
- 付款计划与公寓运营支出一键关联

> **页面**：`PAGE-118` 房东付款计划页面（独立页面）  
> **API**：`GET /landlord-payments`（列表）、`POST /landlord-payments`（记录付款）

## 二、运营模块

### 2.1 租客档案管理

**FR-401 租客基础档案管理**

- 维护租客信息：姓名、手机号（唯一标识）、身份证号、身份证照片（正反面）
- 紧急联系人：姓名、关系、电话
- 工作单位、职位、收入范围（选填）
- 来源渠道：58同城、豆瓣、贝壳、老带新、中介、其他
- 信用评估：内部评分（基于历史缴费记录、退租检查）

> **页面**：`PAGE-201` 租客列表、`PAGE-202` 租客新增/编辑、`PAGE-203` 租客详情（均为独立页面）  
> **API**：`GET /tenants`（列表）、`POST /tenants`（创建）、`GET /tenants/:id`（详情）、`PUT /tenants/:id`（更新）

**FR-402 同住人管理**

- 一个租约可登记多名同住人
- 同住人信息：姓名、身份证号、手机号、与主租客关系
- 同住人数量限制（按户型）

> **页面**：`PAGE-204` 同住人管理页面（独立页面）  
> **API**：`GET /co-residents`（列表）、`POST /co-residents`（创建）、`DELETE /co-residents/:id`（删除）

**FR-403 租客账户体系管理**

- 预付余额：提前缴纳的房租/杂费余额，可用于抵扣未来账单
- 押金余额：已缴纳押金总额
- 待付总额：所有未结清账单合计
- 净余额：预付余额 - 待付总额（正数表示预存，负数表示欠款）
- 交易流水：每一笔资金变动的时间、类型、金额、关联单据、操作人
- 账户调账：支持正负调账，需填写原因，生成审计记录

> **页面**：
>
> - `PAGE-205` 租客交易流水（内嵌于 `PAGE-203` 租客详情页的组件 `TenantTransactionSection`）
> - `PAGE-206` 账户调账（内嵌于 `PAGE-203` 租客详情页的弹窗）  
>   **API**：`GET /tenants/:id/account`（账户余额）、`GET /tenants/:id/transactions`（交易流水）、`POST /tenants/:id/adjust`（账户调账）  
>   **实现形式**：租客详情页「账户余额」区块展示余额，「交易流水」区块展示分页表格，「调账」按钮打开调账弹窗，均无独立路由页面

### 2.2 租约管理

**FR-501 租约签订**

- 必填信息：房间、租客、租期起止、租金周期（月/季/年）、租金金额
- 账单日设置：每月几号出账
- 水电单价设置（可覆盖公寓默认值）
- 押金规则：押N付M
- 附加费用：管理费、卫生费、网络费等（名称、金额可自定义）
- 免租期设置（可选）
- 自动续约开关
- 滞纳金规则：逾期天数 × 日费率
- 签约时自动生成押金账单和首月预付账单
- 签约后房间状态自动变为已入住
- 签约完成通知租客

> **页面**：`PAGE-208` 租约签订向导（独立页面，路径 `/rooms/:id/lease/new`）  
> **API**：`POST /leases`（创建租约）

**FR-502 租约状态机管理**

- 状态定义：草稿 → 待生效 → 生效中 → 即将到期（到期前30天自动标记）→ 退租结算中 → 已退租 → 到期关闭
- 约束规则：同一时间一个房间只能有一个生效中或待生效租约；到期自动续约或关闭

> **页面**：`PAGE-207` 租约列表页面（独立页面）  
> **API**：`GET /leases`（列表）、`PUT /leases/:id`（更新状态）

**FR-503 租约变更管理**

- 支持变更：租金金额、押金金额、水电单价、附加费用、账单日
- 每次变更生成变更记录（变更前/后值、变更时间、操作人、原因）
- 变更后账单按新规则生成
- 不允许变更已终止租约

> **页面**：`PAGE-210` 租约变更页面（独立页面，路径 `/rooms/:id/lease/edit`）  
> **API**：`PUT /leases/:id`（更新租约）

**FR-504 续租管理**

- 到期前30天系统提醒运营人员联系租客
- 续租时选择新租期、新租金（可修改），生成新的租约周期
- 续租记录关联原租约
- 支持不涨价续租/涨价续租
- 续租后原租约状态变为已续租，新租约继承房间

> **页面**：`PAGE-211` 续租页面（独立页面，路径 `/leases/:id/renew`）  
> **API**：`POST /leases/:id/renew`（续租）

**FR-505 换房管理**

- 支持租客在租期内更换房间
- 换房流程：退租结算（原房间）→ 新签约（新房间）
- 换房时押金可转结，租金按实际入住天数计算
- 换房记录关联原租约和新租约

> **页面**：`PAGE-212` 换房页面（独立页面，路径 `/leases/:id/room-change`）  
> **API**：`POST /leases/:id/room-change`（换房）

**FR-506 退租结算管理**

- 退租类型：到期退租、协商退租、违约退租、强制清退
- 结算项目：租金调整、水电结算、押金处理、违约金、赔偿金、其他费用
- 结算流程：发起退租 → 退租检查（拍照）→ 费用核算 → 双方确认 → 资金收付 → 完成
- 退租完成后结清账单、更新租约状态、更新房间状态、更新租客信用记录

> **页面**：`PAGE-213` 退租结算页面（独立页面，路径 `/rooms/:id/lease/terminate`）  
> **API**：`POST /leases/:id/terminate`（退租结算）

### 2.3 水电表具管理

**FR-601 表具档案管理**

- 维护表具信息：编号、类型（水/电/气）、安装位置（房间/公共区域）
- 初始读数、安装日期
- 状态：在用/已拆除/已更换
- 支持父子表关系（总分表核算）

> **页面**：`PAGE-214` 表具列表、`PAGE-215` 表具新增/编辑、`PAGE-216` 表具详情（均为独立页面）  
> **API**：`GET /meters`（列表）、`POST /meters`（创建）、`GET /meters/:id`（详情）、`PUT /meters/:id`（更新）、`DELETE /meters/:id`（删除）

**FR-602 抄表管理**

- 抄表方式：人工录入、批量导入、智能表对接（预留）
- 抄表类型：月度例行、入住抄表、退租抄表、临期抄表
- 读数校验：本期读数小于上期读数时告警
- 用量异常检测：用量超过历史均值200%时标记疑似异常
- 抄表记录关联账单

> **页面**：`PAGE-218` 抄表录入、`PAGE-219` 抄表批量导入、`PAGE-220` 抄表记录（均为独立页面）  
> **API**：`GET /meter-readings`（列表）、`POST /meter-readings`（录入）、`POST /meter-readings/import`（导入）  
> **注意**：后端无独立 `meterReadings.ts` 路由文件，抄表 API 耦合在 `meters.ts` 中

**FR-603 水电计费管理**

- 支持固定单价、阶梯单价（预留）
- 总分表差额自动公摊（按面积/按人头）

## 三、财务模块

### 3.1 账单管理

**FR-701 账单生成引擎**

- 预付账单（房租+杂费）：在账单日自动生成，账单周期为下一个缴费周期
- 后付账单（水电）：在账单日生成待出账单，待抄表后完成出账
- 押金账单：签约时生成
- 结算账单：退租时生成
- 支持手动生成/补发账单

> **实现**：后端定时任务（`services/scheduler.ts`、`services/billing.ts`）自动触发  
> **API**：`POST /bills/generate`（手动生成）、定时任务自动调用

**FR-702 账单生命周期管理**

- 状态定义：草稿 → 待确认 → 待收 → 部分收款 → 已结清 → 已逾期 → 已核销 → 已作废
- 账单到期前3天、逾期当天自动提醒租客
- 账单逾期自动标记：超过到期日+宽限期后变为逾期

> **页面**：`PAGE-301` 账单列表（独立页面）  
> **API**：`GET /bills`（列表）、`PUT /bills/:id`（更新状态）  
> **实现**：逾期状态由后端定时任务自动计算更新

**FR-703 账单子项管理**

- 每个账单包含多个子项：房租、水费、电费、管理费、卫生费等
- 子项独立状态：未收/已收/失败/作废
- 子项支持部分收款

> **页面**：`PAGE-302` 账单详情页面（独立页面，含子项编辑）  
> **API**：`GET /bills/:id`（详情）、`PUT /bills/:id/items/:itemId`（更新子项）

**FR-704 滞纳金自动计算**

- 条件：账单状态为待收/部分收款/已逾期，且超过到期日+宽限期
- 计算：滞纳金 = 未收金额 × 日费率 × 逾期天数
- 每日自动计算，生成滞纳金子项
- 滞纳金可减免（需审批记录）

> **实现**：后端定时任务（`services/overdue.ts`）每日自动计算  
> **API**：由 `GET /bills` 列表接口返回已计算的滞纳金数据  
> **页面**：`PAGE-301` 账单列表、`PAGE-302` 账单详情（展示滞纳金子项）

**FR-705 账单操作管理**

- 收款：支持一次收多笔账单，自动按账龄顺序核销（先老后新）
- 部分收款：账单状态变为部分收款，继续催收剩余
- 核销：长期无法收回的坏账，经审批后核销
- 作废：未收款账单可作废（需原因）
- 拆分：部分收款后将剩余金额拆分为新账单
- 合并：同一租客的多笔账单可合并收款

> **页面**：`PAGE-301` 账单列表（操作入口）、`PAGE-303` 收款登记页面（独立页面）  
> **API**：`POST /bills/:id/pay`（收款）、`PUT /bills/:id/void`（作废）、`PUT /bills/:id/write-off`（核销）  
> **注意**：收款拆分、账单合并功能当前由收款登记页面统一处理，无独立拆分/合并页面

### 3.2 收款管理

**FR-801 收款登记**

- 收款方式：现金、银行转账、微信、支付宝、POS机、抵扣余额、其他
- 收款时可关联具体账单，也可作为预付款存入租客账户
- 支持一笔收款拆分到多笔账单
- 收款后自动生成出纳日记账记录

> **页面**：`PAGE-303` 收款登记页面（独立页面）  
> **API**：`POST /bills/:id/pay`（收款）、`POST /payments`（通用收款，如存在）  
> **注意**：后端无独立 `payments.ts` 路由文件，收款 API 耦合在 `bills.ts` 中

**FR-802 收款核销规则**

- 默认按账龄优先：先核销最早到期的账单
- 支持指定核销某笔账单
- 超额收款存入预付余额

> **页面**：`PAGE-303` 收款登记页面（独立页面，核销规则内嵌于收款流程）  
> **API**：`POST /bills/:id/pay`（收款时自动核销）  
> **实现**：核销逻辑由后端 `services/billingPayments.ts` 处理

**FR-803 退款管理**

- 退押金、退预付款、退多收款
- 退款需审批流程（大额退款需二次确认）
- 退款后更新租客账户余额

> **页面**：`PAGE-304` 退款申请（独立页面）、`PAGE-305` 退款审批（独立页面）  
> **API**：当前前端为演示实现，**后端缺失独立 `refunds.ts` 路由文件**，需补充 `POST /refunds`（申请）、`PUT /refunds/:id/approve`（审批）等端点

**FR-804 发票管理**

- 收款后可申请开票
- 发票信息：抬头、税号、金额、内容、邮寄地址
- 发票状态：待开/已开/已寄出/已签收
- 电子发票支持（预留）

> **页面**：`PAGE-306` 发票管理页面（独立页面）  
> **API**：`GET /invoices`（列表）、`POST /invoices`（申请）、`PUT /invoices/:id`（更新状态）

### 3.3 押金管理

**FR-901 押金全生命周期管理**

- 状态定义：待收 → 已收 → 部分退还 → 全额退还（或已抵扣）

**FR-902 押金统计**

### 3.4 出纳日记账

**FR-1001 统一资金流水管理**

- 字段：日期、收支类型（收/支）、科目分类、金额、支付方式、对方（租客/房东/供应商）、关联单据（账单号/合同号）、摘要、操作人

## 四、系统与权限模块

### 4.1 权限管理

**FR-1201 角色权限管理**

- 系统预置角色：超级管理员、组织所有者、财务专员、运营管家、只读用户
- 支持自定义角色，按功能模块粒度授权

> **API**：`GET /admin/roles`（列表）、`PUT /admin/roles/:id`（更新权限）  
> **注意**：当前仅有平台运营后台版本，**组织内角色权限管理页面缺失**（非 `ops` 路径下的独立页面）

**FR-1202 数据隔离**

- 组织级别隔离：用户只能看到自己组织的数据
- 公寓级别隔离（可选）：管家只能看到自己负责的公寓

> **实现**：`AuthMiddleware`（后端中间件，校验 `x-organization-id` Header）  
> **API**：所有业务 API 均通过 `organizationId` 进行数据过滤

### 4.2 审计日志

**FR-1301 操作审计**

- 记录所有数据变更：操作人、时间、IP、操作类型、变更字段、旧值、新值
- 敏感操作标记：收款、退款、调账、退租结算、作废账单、删除记录
- 审计日志不可删除、不可修改

> **页面**：`PAGE-402` 审计日志页面（独立页面）  
> **API**：`GET /audit-logs`（列表）

### 4.3 数据一致性约束

**FR-1401 核心业务约束**

- C1：同一时间一个房间只能有一个生效中或待生效租约
- C2：账单金额 = 各子项金额之和
- C3：租客账户净余额 = 预付余额 + 押金余额 - 待付总额
- C4：押金已收 ≥ 已退 + 已抵扣
- C5：账单已收 ≤ 账单总额
- C6：租约状态变更必须按状态机规则
- C7：房间状态变更必须关联对应业务单据
- C8：已收款账单不可删除

> **实现**：数据库层（Prisma Schema 约束 + 事务）+ 应用层（Service 层校验）

**FR-1402 数据校验**

- 租约日期不可重叠
- 水电读数必须递增
- 金额字段必须 ≥ 0
- 必填字段校验

> **实现**：应用层（Zod Schema 校验）+ 数据库层（Prisma Schema 约束）

## 五、非功能性需求

### 5.1 性能

- 列表查询支持分页，默认20条/页
- 报表查询最大时间范围12个月
- 定时任务不影响正常业务响应

> **实现**：所有列表 API 均支持 `page`/`pageSize` 参数；报表查询限制由前端组件控制

### 5.2 安全

- 敏感操作需二次确认
- 金额修改需填写原因
- 数据删除为软删除
- 审计日志永久保留

> **实现**：敏感操作由前端 `Modal.confirm` 二次确认；金额修改强制填写 `reason` 字段；数据库采用 `deletedAt` 软删除标记

### 5.3 可靠性

- 定时任务支持失败重试（最多3次）
- 账单生成支持手动补发
- 数据校验在应用层和数据库层双重保障

> **实现**：后端定时任务（`services/scheduler.ts`）含重试逻辑；`POST /bills/generate` 支持手动补发
