# Tenant Hub 核心业务验收脚本

> 目标：在发布前用一套固定流程验证 Tenant Hub 的核心业务闭环。每轮验收使用全新的测试账号、测试组织和测试房源，避免历史数据干扰。

## 0. 验收准备

### 环境

- API 健康检查：`http://localhost:4000/health`
- API 基础地址：`http://localhost:4000/api`
- 运营端：`http://localhost:5173`
- 移动端本地开发：`cd apps/mobile && pnpm web`（验收脚本会自动启动，无需手动操作）

## 1. Docker 启动验收

### 1.1 启动服务

运行：

```bash
docker compose up --build -d
```

预期：

- `tenant-hub-postgres` 为 `healthy`
- `tenant-hub-api` 为 `healthy`
- `tenant-hub-ops-web` 为 `running`

### 1.2 检查容器状态

运行：

```bash
docker compose ps
```

预期：

- `postgres` 端口映射为 `5433->5432`
- `api` 端口映射为 `4000->4000`
- `ops-web` 端口映射为 `5173->80`

### 1.3 检查 API 健康

运行：

```bash
curl -fsS http://localhost:4000/health
```

预期：

```json
{"ok":true}
```

### 1.4 检查前端入口

运行：

```bash
curl -I http://localhost:5173
```

预期：

- 运营端返回 `HTTP/1.1 200 OK`

## 2. 账号与组织验收

### 2.1 注册业主账号

步骤：

1. 打开移动端 Web 预览（如未运行，执行 `cd apps/mobile && pnpm web`）。
2. 切换到注册。
3. 输入业主手机号、用户名、密码、确认密码。
4. 点击获取验证码。
5. 从 API 容器日志读取验证码：

```bash
docker logs tenant-hub-api --tail=80
```

6. 输入验证码并提交注册。

预期：

- 注册成功后进入移动端应用。
- `个人/设置` 页面能看到当前用户信息。
- 刷新页面后登录态可恢复或能重新登录。

### 2.2 创建组织

步骤：

1. 进入组织设置或首次登录组织引导。
2. 创建组织 `验收测试组织`。
3. 填写描述 `发布前核心验收使用`。

预期：

- 创建成功后页面显示当前组织。
- 后续公寓、房间、租约、账单页面不再提示“请先选择组织”。

### 2.3 注册成员并加入组织

步骤：

1. 使用成员手机号注册第二个账号。
2. 复制业主组织的邀请码或组织编码。
3. 成员账号加入该组织。
4. 业主账号进入成员管理查看成员。

预期：

- 新成员默认角色为只读成员。
- 业主能修改成员角色。
- 只读成员不能执行新增公寓、签约、收款等管理操作。

### 2.4 所有者转移

步骤：

1. 业主账号将组织所有者转移给成员账号。
2. 成员账号刷新组织成员列表。
3. 原业主账号尝试执行仅所有者可做的操作。

预期：

- 成员成为所有者。
- 原业主降级为管家或非所有者角色。
- 非所有者不能删除组织。

## 3. 公寓与房间验收

### 3.1 创建公寓

步骤：

1. 进入公寓页面。
2. 新增公寓 `云杉公寓`。
3. 填写地址、楼层、水价 `4`、电价 `0.8`。
4. 填写房东合同信息。

预期：

- 公寓列表出现 `云杉公寓`。
- 公寓详情能看到水电单价和合同信息。

### 3.2 配置可选费用

步骤：

1. 在公寓详情新增费用项目 `物业费`，金额 `100`。
2. 新增费用项目 `网络费`，金额 `80`。
3. 禁用其中一个费用项目后再重新启用。

预期：

- 启用的费用项目在签约表单中可选。
- 禁用的费用项目不应作为可选费用出现。

### 3.3 批量创建房间

步骤：

1. 在公寓页使用批量创建房间。
2. 生成 `201`、`202`、`203`。
3. 提交创建。
4. 进入房间页查看。

预期：

- 三个房间均显示为空闲。
- 房间排序稳定。
- 重复提交相同房间不会创建重复房间。

## 4. 租约生命周期验收

### 4.1 空房签约

步骤：

1. 在房间 `201` 上点击签约。
2. 填写租客 `张三`、手机号 `13900002001`。
3. 租期选择本月 5 日到 6 个月后的 4 日。
4. 周期选择月付。
5. 月租 `3000`，押金 `3000`，水价 `4`，电价 `0.8`。
6. 勾选 `物业费`。
7. 提交签约。

预期：

- 房间 `201` 状态变为已租。
- 租约列表出现该租约。
- 系统自动生成首期账单。

### 4.2 非空房签约拦截

步骤：

1. 再次尝试对房间 `201` 签约。

预期：

- 系统拒绝签约。
- 错误提示表达“仅空闲房间可以签约”或同等含义。

### 4.3 到期/协商退租

步骤：

1. 对 `201` 的租约执行协商解约。
2. 填写退租日期和原因。
3. 返回房间列表。

预期：

- 租约状态为已终止。
- 房间 `201` 恢复为空闲。
- 已有账单不会丢失。

### 4.4 到期解约日期校验

步骤：

1. 新签一条租约。
2. 选择到期解约。
3. 退租日期填写早于原租约结束日期。

预期：

- 系统拒绝提交。
- 错误提示表达“到期解约的退租日期不能早于原租约结束日期”或同等含义。

## 5. 账单与收款验收

### 5.1 查看周期账单

步骤：

1. 进入账单页。
2. 查看签约后生成的周期账单。

预期：

- 能看到租客、房间、账期、应收金额、账单状态。
- 首期账单包含房租和押金。
- 可选费用出现在账单明细中。

### 5.2 水电读数录入

步骤：

1. 找到账单中的水电项。
2. 填写上期水表 `10`、本期水表 `18`。
3. 填写上期电表 `100`、本期电表 `160`。
4. 提交。

预期：

- 水电金额为 `80`。
- 账单总额随水电金额刷新。
- 若本期读数小于上期读数，系统拒绝提交。

### 5.3 部分收款和全额收款

步骤：

1. 对周期账单收款 `1000`，方式 `微信`。
2. 查看账单状态。
3. 再收剩余金额。
4. 查看账单状态。

预期：

- 第一次收款后状态为部分收款。
- 收满后状态为已支付。
- 收款记录保留操作人、时间、金额、方式。

### 5.4 水电导出与导入

步骤：

1. 进入水电待处理或账单导出入口。
2. 导出待录入水电账单。
3. 按导出格式填写读数。
4. 导入读数。

预期：

- 导出内容包含账单 ID、房间号、租客、账期。
- 导入后对应水电项更新金额。
- 错误格式或缺失账单项时系统给出失败结果。

## 6. 运营端验收

### 6.1 首个管理员进入运营端

步骤：

1. 打开 `http://localhost:5173`。
2. 使用业主账号登录。

预期：

- 当系统没有任何平台管理员时，首个登录用户可进入运营端。
- 能看到运营概览、用户、组织、套餐、角色、系统参数入口。

### 6.2 平台权限管理

步骤：

1. 在用户管理中搜索成员账号。
2. 授予成员账号 `OPERATOR` 或 `ADMIN`。
3. 成员账号登录运营端。
4. 再移除成员账号运营权限。

预期：

- 授权后成员能进入运营端。
- 移除后成员再次进入运营端被拒绝。

### 6.3 套餐与额度包

步骤：

1. 新建套餐 `验收基础版`。
2. 设置公寓、房间、成员额度和价格。
3. 给测试组织发放额度包。
4. 回到移动端组织设置查看订阅和额度。

预期：

- 套餐创建后可在组织套餐中查看。
- 额度包发放后组织额度增加。

### 6.4 角色权限配置

步骤：

1. 在运营端新建自定义角色 `验收只收款`。
2. 仅勾选账单查看和账单管理权限。
3. 将组织成员切换到该角色。
4. 成员登录移动端。

预期：

- 成员能查看账单并收款。
- 成员不能新增公寓、编辑房间、签约租约。

## 7. 发布判定

### 必须通过

- Docker 启动验收全部通过。
- 账号、组织、公寓、房间、租约、账单、收款主流程全部通过。
- 权限边界没有 P0/P1 越权问题。
- `pnpm test`、`pnpm typecheck`、`pnpm lint`、`pnpm build` 全部通过。

### 可带着发布但需记录

- 非核心页面文案问题。
- ops-web 构建 chunk 超过 500 kB 的性能提醒。
- 不影响数据正确性的移动端样式瑕疵。

### 不可发布

- Docker 无法冷启动。
- 数据库迁移失败。
- 组织间数据可互相访问。
- 已租房间可重复签约。
- 账单金额错误。
- 普通用户可进入运营端或修改平台配置。
